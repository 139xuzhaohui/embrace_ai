#### rag （Retrieval Augmented Generation）- 检索增强生成 【主要的几步 索引、检索 还有啥吗？？】
1. LLM 的缺陷 （不及时、 垂类信息不匹配、 幻觉）
2. 为什么要用rag： 1、提高准确性 2、减少训练成果（晨本）、 3、适应性强-快速适配新资料
3. RAG 使用的时机-是什么时候呢？ 是穿插的吗？还是？ 穿插的很多种   
> 一个标准 RAG pipeline 通常是： **split后先向量化再去索引+检索**
> - Load（加载）：读取 PDF/网页/数据库/飞书/Notion 等内容
> - Split（切分）：按 chunk 切片（长度、重叠、按标题等策略）
> - Embed（向量化）：对每个 chunk 计算 embedding 向量
> - Index（建索引）：将向量 + 文本 + 元数据写入向量数据库/索引结构
> - Retrieve（检索）：对用户 query 做 embedding → 相似度搜索 top-k
> - Rerank（重排，可选但很常用）：**用更强模型对 top-k 做精排**，减少“看起来相似但不回答”的片段
> - Augment（组装上下文）：**把最相关 chunk 拼成上下文（含引用）**
> - Generate（生成）：LLM 基于“问题 + 检索内容 + 指令”生成答案
> - Cite/Evaluate（引用与评估，可选）：输出引用、做命中率/幻觉率评估

6. RAG vs Fine-tuning
- RAG（检索增强生成）是把内部的文档数据先进行embedding，借助检索先获得大致的知识范围答案，再结合prompt给到LLM，让LLM生成最终的答案 【 那就是 在llm接受之前 拿到rag的内容？？？】llm 会结合rag后的内容
- Fine-tuning（微调）是用一定量的数据集对LLM进行局部参数的调整，以期望LLM更加理解我们的业务逻辑，有更好的zero-shot能力。 【微调的对象是？】
-  微调的对象：1、**全量微调**：更新模型大部分/全部参数  2、 **局部PEFT/LoRA**：主要更新“新增的小参数模块”（但依然是在训练阶段改变模型的可学习参数）

> 更适合：
> RAG
> 1.改的是外部知识获取方式：把文档作为“证据”塞进上下文。
> 2.模型参数不变（除非你同时做微调）。
> 3.适合：��识频繁更新、需要引用、私有文档问答。   

> 微调 适合：
> 1. 固定任务模式（客服话术、结构化输出、特定流程）
> 2. 希望模型“不检索也能按规则做事”
> 3. 需要更稳定的输出格式/风格 【这里得区别响应？？？】

#### 预训练 && 微调
> 预训练（Pre-training）：用海量通用数据，**让模型学通用语言/世界知识（成本极高，通常一次成型为“基座模型”）**。
> 微调（Fine-tuning）：**在“已有基座模型”上**，用更小、更针对性的数据继续训练，让它更符合你的任务/领域/风格。
> 但从“训练动作”角度看，**它们都属于“继续用梯度下降更新参数”的训练过程**，  **只是数据与目标不同**
> 预训练：学通用能力
> 微调：学专项能力/行为规范

####  微调 vs prompt/角色定义/JSON 输出（你觉得像 function calling）

> - Prompt/角色定义：不改参数，只靠上下文约束。优点是快、便宜；缺点是不稳定，**复杂时容易跑偏**。   
> - Function calling / JSON schema 约束：更像“推理时的输出约束与接口协议”，很多平台是通过解码约束/结构化提示实现的。它能显著提高结构化输出成功率，**但仍然不等于模型“真的懂业务规则”**。   
> - 微调：把你想要的行为“写进参数里”。优点是更稳定、更省prompt长度、**更像内化习惯**；缺点是需要数据、训练、评估与版本管理。   
> - 结论：
> 1. “要求输出 JSON”这种格式类需求：通常优先用 schema/约束解码 + prompt 就够了，微调是更后面的选项。   
> 2. “遵循复杂业务流程、专有术语解释、固定话术风格、工具选择策略很稳定”这类：**微调更有价值**。

## Agents是什么
大语言模型可以接受输入，可以分析&推理、可以输出文字、代码、媒体。   
然而，其无法像人类一样，**拥有规划思考能力、运用各种工具与物理世界互动，以及拥有人类的记忆能力**。   
Al Agents是基于LLM的能够 **自主理解、自主规划决策、执行复杂任务的智能体**。   
Agent的设计目的是为了处理那些简单的语言模型可能无法直接解决的问题，尤其是当这些任务涉及到多个步骤或者需要外部数据源的情况。
